# AGENTS.md — инструкции для Codex (AI-Brain)

## Кто пользователь и что это за репозиторий
Пользователь: Виталий Семьянов — преподаватель бачаты, продюсер онлайн-запусков, директор по маркетингу D'Arky Dance (Новосибирск).
Этот репозиторий — система управления проектами: каждая папка = отдельный проект.

## Структура проектов
В каждом проекте (папке) обычно есть:
- brief.md — контекст проекта
- decisions.md — принятые решения (НЕ противоречить)
- log.md — текущий этап и история
Дополнительно по задачам ведём:
- research.md — исследование (append-only)
- plan.md — план + TODO (append-only)

## Дефолтный порядок работы (ОБЯЗАТЕЛЬНО)
Всегда следуй порядку: **Research → Plan → STOP → (только после “GO” — выполнение)**

### 1) Перед выполнением любой задачи
1. Определи, к какому проекту относится задача (какая папка).
2. Прочитай в папке проекта:
   - brief.md
   - decisions.md
   - log.md

3. Сделай DEEP RESEARCH и запиши в `<project>/research.md` (создай файл, если его нет).
   Формат: добавляй новую секцию В КОНЕЦ файла:
   - дата + название задачи
   - что уже есть в проекте по теме задачи (файлы/разделы)
   - важные ограничения/конвенции (“не трогать”, стиль, решения)
   - риски/подводные камни
   - максимум 3–5 вопросов к пользователю, если без ответов план будет слабым

4. Сделай PLAN и запиши в `<project>/plan.md` (создай файл, если его нет).
   Формат: добавляй новую секцию В КОНЕЦ файла:
   - дата + название задачи
   - цель
   - критерии готовности (acceptance criteria)
   - пошаговый план (что и где меняем)
   - список файлов, которые будем менять/создавать
   - TODO-чеклист:
     - [ ] пункт 1
     - [ ] пункт 2
   - “Не делаем” (границы scope, чтобы не расползлось)

5. **STOP**: не вноси изменения в другие файлы и не выполняй команды, пока пользователь не напишет **GO**.
   Исключение: если пользователь прямо просит “делай сразу” или пишет GO в первом сообщении — можно продолжать, но всё равно сначала должен быть plan.md (хотя бы краткий).

### 2) После команды “GO”
1. Выполняй строго по plan.md, избегая лишних улучшений “заодно”.
2. Отмечай выполненные пункты TODO как [x].
3. По завершении:
   - обнови `<project>/log.md`: дата + кратко что сделано
   - если нужно решение пользователя: пометь в log.md как **[РЕШЕНИЕ ТРЕБУЕТСЯ]**
   - допиши в конце секции plan.md короткий итог + как проверить результат (шаги)

## Правила редактирования файлов
- Редактируй существующие файлы, не создавай дубликаты.
- Не удаляй существующий контент — только дополняй.
- Не меняй decisions.md и brief.md без явной команды пользователя.
- Новый файл создавай только если:
  1) нет подходящего существующего, и
  2) он указан в plan.md, и
  3) название/путь логичны для структуры проекта.

## Если не хватает информации
- Не додумывай. Задай конкретные вопросы (до 5).
- Если можно — сделай план на допущениях, но явно перечисли Assumptions в plan.md.

## Если есть противоречия между файлами
- Остановись и перечисли: какие файлы и в чём именно противоречат.
- Не решай сам, какой файл “правильный”.

## Язык и стиль
- Пиши и редактируй файлы на русском.
- Стиль: конкретно, без воды, с примерами, короткими списками.
- Не выдумывай факты/цифры/обещания — если данных нет, спроси.

## AGENTS Harness v2 (добавление от 2026-03-05)

> Если правила выше и ниже конфликтуют, приоритет у этого блока `AGENTS Harness v2`.

### Приоритет инструкций
1. Локальный `AGENTS.md` в папке проекта имеет приоритет над корневым в пределах своей папки.
2. Корневой `AGENTS.md` — базовый стандарт для всего `ai-brain`.
3. При конфликте правил агент обязан остановиться и запросить выбор пользователя.

### Non-mutating vs Mutating (до GO)
До команды `GO` разрешены только non-mutating действия для исследования:
- чтение файлов, поиск, анализ структуры,
- запуск проверок без изменения tracked-файлов,
- dry-run команды, которые не меняют содержимое репозитория.

До `GO` запрещены mutating действия:
- редактирование/создание/удаление tracked-файлов,
- массовые rewrite-операции,
- запуск команд, которые изменяют tracked-файлы.

Исключение прежнее: если пользователь явно просит делать сразу или пишет `GO` в первом сообщении, можно выполнять, но после фиксации плана.

### Контракт Task ID (context graph lite)
Для каждой новой задачи обязателен `Task ID` (формат: `HARNESS-YYYY-MM-DD-XX` или проектный аналог).

#### Обязательный минимум в `research.md`
- `Task ID`
- `Контекст задачи`
- `Ограничения`
- `Риски`
- `Вопросы/Assumptions`
- `Источники (если были внешние)`

#### Обязательный минимум в `plan.md`
- `Task ID`
- `Цель`
- `Acceptance criteria`
- `Пошаговый план`
- `Файлы`
- `TODO`
- `Не делаем`
- `Итог проверки`

#### Обязательный минимум в `log.md`
- `Task ID`
- `Что сделано`
- `Что проверить`
- `[РЕШЕНИЕ ТРЕБУЕТСЯ]` при необходимости решения от пользователя

`Task ID` в `research.md`, `plan.md` и `log.md` должен совпадать для одной задачи.

### Канон и производные AGENTS
- Канонические файлы: корневой `AGENTS.md` и проектные `AGENTS.md`.
- Производные файлы: `.claude/worktrees/*/AGENTS.md`.
- Производные файлы не редактируются вручную; синхронизация выполняется скриптом.

### Harness-команды (базовый уровень)
- Проверка быстрым режимом: `python3 scripts/harness_check.py --mode fast`
- Проверка строгим режимом: `python3 scripts/harness_check.py --mode strict`
- Проверка синхронизации AGENTS: `python3 scripts/harness_sync_agents.py --check`
- Синхронизация производных AGENTS: `python3 scripts/harness_sync_agents.py --write`

### Optional pre-commit
- Hook поддерживается как opt-in.
- Включение через команду: `make harness-hook-install`.

### Maintenance cadence
- Раз в месяц делать мини-аудит процесса (10–20 минут):
  - проверить актуальность AGENTS,
  - прогнать `harness_check` и `harness_sync_agents --check`,
  - зафиксировать вывод в `agent-harness/changelog.md`.
