## Правила работы

### Базовый порядок работы (по умолчанию всегда так)

**Research → Plan → STOP → (только после команды “GO” — выполнение)**

Цель: сначала понять контекст и согласовать план, и только потом вносить изменения (код/тексты/структуру файлов).

- **Research**: кратко и конкретно зафиксировать понимание проекта/задачи и текущего состояния.
- **Plan**: расписать шаги, файлы, критерии готовности и TODO.
- **STOP**: остановиться и дождаться команды пользователя **“GO”**.
- **GO**: выполнить план, отмечая TODO и обновив log.md.

> Исключение: если пользователь прямо пишет “GO” в первом сообщении или просит “делай сразу”, можно перейти к выполнению без паузы, но всё равно сначала сделать мини-план (хотя бы 5–10 строк) в plan.md.


### Перед выполнением любой задачи:
 
1. Прочитай этот файл (CLAUDE.md)
2. Прочитай `brief.md` нужного проекта — пойми контекст
3. Прочитай `decisions.md` — не противоречь принятым решениям
4. Проверь `log.md` — пойми, на каком мы этапе

5. **Сделай Research и запиши в файл проекта `research.md`** (создай, если его нет):
   - что уже есть в проекте по теме задачи (файлы/разделы/артефакты)
   - важные ограничения (решения, стиль, формат, “не трогать”)
   - риски/подводные камни
   - максимум 3–5 вопросов ко мне, если без ответов нельзя сделать хороший план  
   Формат: добавляй новую секцию в конец файла с датой и названием задачи.

6. **Сделай Plan и запиши в файл проекта `plan.md`** (создай, если его нет):
   - цель задачи
   - критерии готовности (acceptance criteria)
   - пошаговый план (что и где меняем)
   - список файлов, которые будешь менять/создавать
   - TODO-чеклист вида:
     - [ ] пункт 1
     - [ ] пункт 2
   - “Не делаем” (границы scope, чтобы не расползлось)
   Формат: добавляй новую секцию в конец файла с датой и названием задачи.

7. **Остановись. Не вноси изменения в другие файлы, пока пользователь не напишет “GO”.**
 
 
### После выполнения задачи:
 
1. Обновляй `log.md` проекта с датой и кратким описанием того, что было сделано
2. Если нужно моё решение — отметь в `log.md` как **[РЕШЕНИЕ ТРЕБУЕТСЯ]**
3. В `plan.md` отметь выполненные пункты TODO (поставь [x]) и допиши короткий итог по этой секции
 
 
### При работе с файлами:
 
1. Редактируй существующие файлы — не создавай дубликаты
2. Не удаляй существующий контент, только дополняй
3. Сохраняй результат в соответствующий `.md` файл проекта
4. Новый файл создавай только если:
   - нет подходящего существующего файла, и
   - это явно требуется по плану (plan.md), и
   - имя/место файла логично в структуре проекта
 
### D'Arky Dance: SMM / тексты:

Все правила по написанию постов и SMM для D'Arky лежат в `darky-dance/smm/CLAUDE.md` (читать перед написанием).
 
### Чего не делать:
 
- Не выдумывай факты обо мне или моих проектах
- Не меняй `decisions.md` без моей явной команды
- Не меняй `brief.md` без моей явной команды
- Не додумывай — если нужна информация от меня, спроси напрямую
- Не начинай “реализацию” (правки любых файлов кроме research.md/plan.md), пока нет команды **GO**
 
 
### Если видишь противоречие между файлами:
 
- Останови работу и сообщи мне, какие именно файлы и в чём противоречат
- Не принимай решение сам, какой файл «правильный»
 
 
## Формат
 
- Все файлы — markdown
- Язык — русский
- Стиль — конкретный, без воды, с примерами
- Если не хватает информации — скажи, что именно нужно узнать
 
 
## Мой стиль
 
- Простой язык, без корпоративного буллшита
- Конкретные действия, а не абстрактные советы
- Если нужна информация от меня — спроси, не додумывай
