# AGENTS.md — D'Arky Dance / SMM

## Контекст
Эта папка — SMM-контент для танцевальной школы D'Arky Dance.
Цель: писать тексты в фирменном стиле, без выдуманных фактов, с понятным CTA.

## Перед написанием любого поста / SMM-текста (ОБЯЗАТЕЛЬНО)
1) Прочитай `darky-dance/smm/copywriting-rules.md`
2) Прочитай `darky-dance/brand-voice.md` (секция «Голос в соцсетях»)
3) При необходимости — используй референсы:
   - `darky-dance/smm/reference-texts.md`
   - `darky-dance/smm/style-analysis.md`
4) Выбери шаблон из `darky-dance/smm/templates/` под нужный тип поста

## Порядок работы (как в корне)
Всегда: **Research → Plan → STOP → (после “GO” — выполнение)**

### Research (внутри проекта)
- Записывай в `darky-dance/research.md` (append-only, новая секция в конец):
  - дата + тема поста/задачи
  - для кого текст и какой CTA
  - какие данные/факты есть в файлах, а каких не хватает
  - максимум 3–5 вопросов к пользователю, если без ответов нельзя

### Plan
- Записывай в `darky-dance/plan.md` (append-only, новая секция в конец):
  - цель поста (зачем пишем)
  - критерии готовности (что значит “хорошо”)
  - структура (хуки → смысл → доказательства → CTA)
  - 2–5 вариантов хуков (если уместно)
  - TODO-чеклист [ ]

### STOP
- Не вноси изменения в контент-файлы (черновики/планы/посты), пока пользователь не напишет **GO**.
- Исключение: если пользователь прямо просит “делай сразу” или пишет GO в первом сообщении — можно продолжать, но всё равно сначала должен быть plan.md (хотя бы краткий).

## Правила качества (жёстко)
- Не выдумывай факты, цифры, цены, расписание, адреса, условия — если данных нет в файлах, спроси.
- Не меняй `brief.md` и `decisions.md` без явной команды пользователя.
- Сохраняй стиль: простой язык, конкретика, без корпоративщины, с примерами.
- Один пост = один главный CTA (не распыляй на несколько действий).
- Не пиши “водянистые” вводные — сразу к смыслу.

## Сохранение результата
- Не создавай дубликаты файлов. Редактируй существующие.
- Не удаляй старый контент — только дополняй.
- Готовые тексты сохраняй в соответствующие файлы внутри `darky-dance/smm/` (план/черновики/контент-план — по назначению).

## Работа с визуалами и image-промтами (Nano Banana Pro)

### Перед генерацией промта для изображения (ОБЯЗАТЕЛЬНО)
1) Прочитай `darky-dance/smm/image-prompts-reference.md`
2) Найди релевантные карточки по типу задачи и тегам
3) Используй их как базу для новых промтов

### Правила генерации
- Генерируй промты только если пользователь явно попросил визуал/картинку.
- На один запрос выдавай 3 варианта промта.
- Язык промтов: EN.
- Формат хранения в библиотеке: только Raw.
- Если пользователь не просил визуал, промты автоматически не добавляй.

### Правила пополнения библиотеки
- Новые проверенные промты добавляй в `image-prompts-reference.md` только append-only.
- Не переписывай и не сокращай уже сохранённый `RAW Prompt (EN)`.
- Фиксируй в карточке комментарии по фактическому результату с датой.

## Visual DNA (обязательно для image-промтов)

### Перед генерацией визуального промта
1) Прочитай `darky-dance/smm/visual-style-dna.md`
2) Выбери ветку стиля: `core` (по умолчанию) или `experimental` (только для явной A/B-задачи или прямого запроса)
3) Проверь ротацию по `rotation_tags`, чтобы не повторять один и тот же визуальный паттерн подряд

### Что обязательно зафиксировать в ответе при генерации 3 вариантов
- `style_branch` для каждого варианта
- ключевые `rotation_tags` (layout/accent/motif/density)
- подтверждение, что соблюдены Hard Rules из `visual-style-dna.md`

## AGENTS Harness v2 (добавление от 2026-03-05)

> Этот файл — локальный override для `darky-dance/smm/` и имеет приоритет над корневым `AGENTS.md` внутри этой папки.

### Non-mutating vs Mutating (до GO)
До `GO` разрешены только non-mutating действия:
- чтение и анализ `copywriting-rules.md`, `brand-voice.md`, референсов и шаблонов,
- исследование без изменения контент-файлов.

До `GO` запрещены mutating действия:
- создание/редактирование постов и контент-файлов,
- изменение шаблонов и библиотек промтов,
- любые правки tracked-файлов вне research/plan при отсутствии `GO`.

### Контракт Task ID для SMM-задач
Каждая новая SMM-задача должна иметь `Task ID` и связку между `research.md`, `plan.md`, `log.md`.

#### Минимум в `darky-dance/research.md`
- `Task ID`
- `Контекст задачи`
- `Ограничения`
- `Риски`
- `Вопросы/Assumptions`
- `Источники (если были внешние)`

#### Минимум в `darky-dance/plan.md`
- `Task ID`
- `Цель`
- `Acceptance criteria`
- `Пошаговый план`
- `Файлы`
- `TODO`
- `Не делаем`
- `Итог проверки`

#### Минимум в `darky-dance/log.md`
- `Task ID`
- `Что сделано`
- `Что проверить`
- `[РЕШЕНИЕ ТРЕБУЕТСЯ]` при необходимости

`Task ID` в `darky-dance/research.md`, `darky-dance/plan.md` и `darky-dance/log.md` должен совпадать для одной задачи.

### Применение локальных и глобальных правил
- Сначала применяются правила этого SMM-файла.
- Далее применяются правила корневого `AGENTS.md`, если они не конфликтуют с локальными.
- При конфликте агент обязан остановиться и запросить решение пользователя.

### Проверка процесса (рекомендуется перед завершением задачи)
- `python3 scripts/harness_check.py --mode fast`
- `python3 scripts/harness_check.py --mode strict`
