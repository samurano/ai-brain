### Что уже есть в проекте по теме задачи
- В `darky-dance/smm/image-prompts-reference.md` собраны 16 карточек (`NBP-001`…`NBP-016`) с raw-промтами.
- В `darky-dance/brand-voice.md` зафиксированы визуальные ориентиры бренда: тёмная неоновая палитра и ночной вайб.
- В `darky-dance/smm/AGENTS.md` уже есть процесс генерации промтов, но нет формализованной ДНК стиля и анти-повторной ротации.

### Наблюдения по повторяющимся паттернам (ядро)
- Почти во всех core-карточках повторяется тёмная база: charcoal/near-black, аккуратная глубина, high-contrast читаемость.
- Устойчивый акцентный градиент: `#2EC5FF -> #7A4DFF -> #FF3DBB`.
- Эффекты строго ограничены: glow/bloom на рамках, бордерах, divider/grid; body-текст без glow/outline/shadow.
- Частый фирменный мотив: ECG/heartbeat как фоновый элемент низкой плотности.
- Типографический режим стабилен: чёткая кириллица, чистые гротески (Inter/Manrope/Montserrat), строгие сетки и безопасные поля.
- Для photo-based задач повторяется принцип identity lock и реалистичная интеграция света.

### Наблюдения по вариативности (экспериментальная ветка)
- В `NBP-015` и частично `NBP-016` есть отход от «чистого неона» в сторону editorial/quiet-luxury.
- Даже в альтернативе сохраняются: дисциплина сетки, высокая читаемость, минимализм, отсутствие визуального шума.

### Важные ограничения и конвенции
- Документ ДНК нужен отдельным файлом в `darky-dance/smm/`.
- Модель: `Core + Experimental`.
- Правила задаются как `Hard + Flex matrix`, чтобы сохранять узнаваемость без шаблонности.
- Перед выдачей новых промтов нужен протокол ротации против визуального утомления.

### Риски и подводные камни
- Если не разделить hard/flex, стиль может стать либо однообразным, либо расплывчатым.
- Без явного статуса ветки (`core`/`experimental`) команда может случайно уходить в альтернативу в регулярных публикациях.
- Без rotation-тегов есть риск повторять один и тот же layout и акценты в соседних постах.

### Вопросы к пользователю
- Вопросов нет: данных достаточно для фиксации Visual DNA v1.

### Что такое Aider (по официальным источникам)
- `aider` — open-source AI pair-programming инструмент в терминале, который работает внутри git-репозитория и умеет вносить изменения в файлы через чат.
- По официальному README поддерживает работу с несколькими моделями (OpenAI, Anthropic, Gemini, DeepSeek, локальные модели и др.) и ориентирован на workflow разработчика в CLI.
- По документации поддерживает режимы работы (включая разделение на «архитектора/редактора») и может запускать lint/test в цикле правок.
- В релизах проекта зафиксирована быстрая эволюция; актуальная ветка релизов на момент исследования содержит обновления под GPT-5 (релизная линейка 2026).

### Важные ограничения и конвенции
- Для ваших задач основная ценность Aider появляется в кодовых подпроектах; в чисто контентных и стратегических задачах (SMM/маркетинг/управление в Markdown) прирост ограничен.
- Любое внедрение должно не ломать текущий процесс фиксации контекста (`brief/decisions/log + research/plan`).
- Решение об инструменте должно учитывать не только «качество кода», но и операционную сложность: единые правила, онбординг, стоимость API, контроль данных.

### Риски и подводные камни
- Дублирование агентного стека: параллельное использование Codex + Claude + Aider без роли для каждого инструмента создаёт хаос и разный стиль изменений.
- Стоимость: при больших контекстах и длинных итерациях в CLI расход токенов может заметно вырасти.
- Data/privacy: при работе с внешними LLM нужно отдельно контролировать, какие проектные данные отправляются в облачные модели.
- Низкая окупаемость для не-кодовых задач: внедрение может усложнить процесс там, где уже достаточно текущего подхода.
