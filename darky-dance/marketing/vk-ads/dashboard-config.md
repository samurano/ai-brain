## 4) Определение уровня отчёта (`level`)

Приоритет определения:
1. По имени файла (raw VK):
   - `ads_daily_report*` -> `campaign`
   - `groups_daily_report*` -> `adset`
   - `banners_daily_report*` -> `ad`
2. По имени листа в XLSX:
   - `Кампании` -> `campaign`
   - `Группы` -> `adset`
   - `Объявления` -> `ad`
3. Fallback по колонкам ID:
   - если есть `ID объявления` -> `ad`
   - если есть `ID группы` -> `adset`
   - если есть `ID кампании` -> `campaign`

## 6) VAT (НДС)
- `VAT_RATE = 20%`
- `VAT_COEFF = 1.2`
- `monetary_display = gross`

Правило:
- На листе `Данные` храним `net` из кабинета VK.
- На листах `Сводка` и `Кампании` все рублёвые метрики отображаются как `gross = net * 1.2`.

## 7) KPI-логика в дашборде
- Управленческие KPI на листе `Сводка` считаются только по строкам `level='adset'`.
- Период задаётся вручную:
  - `Сводка!B2` — дата начала
  - `Сводка!B3` — дата конца
- Целевой CPL (gross) хранится в `Сводка!B4`.

## 8) Светофор CPL (gross)
- `CPL_TARGET_GROSS = 1320`
- `CPL_WARN_MAX_GROSS = 1584` (`+20%` к цели)
- `CPL_RED_FROM > 1584`

Интерпретация:
- `🟢` — CPL ниже цели
- `🟡` — CPL в зоне допуска
- `🔴` — CPL выше допустимого порога

## 9) Операционный режим загрузки
- Режим: `full refresh` листа `Данные` (очистка строк под шапкой + полная перезапись).
- Источник: все CSV/XLSX в `darky-dance/marketing/vk-ads/data/` кроме `README.md`.
- Причина: исключить дубли и обеспечить идемпотентность повторных загрузок.

## 10) Блоки листа «Сводка» (актуальная структура)

### Управленческий KPI-блок
- `A2:B4` — параметры периода и целевой CPL (gross).
- `A6:B14` — основные KPI.
- Рублёвые KPI считаются с `ROUND(...;0)`:
  - `B6` расход
  - `B8` CPL
  - `B10` стоимость пришедшего
  - `B13` CAC

### Недельный блок
- Диапазон: `D5:G`
- Колонки:
  - `D` Неделя
  - `E` Расход, ₽ (gross, округление до целых)
  - `F` Заявки
  - `G` CPL, ₽ (gross, округление до целых)
- Источник: только `level='adset'`, фильтр по периоду `B2:B3`.

### Месячный блок (вся история)
- Диапазон: `I5:L`
- Колонки:
  - `I` Месяц (`YYYY-MM`)
  - `J` Потрачено, ₽ (gross, округление до целых)
  - `K` Заявки
  - `L` CPL, ₽ (gross, округление до целых)
- Источник: только `level='adset'`, без ограничения периода `B2:B3`.

### Доп. KPI
- Блок: `A16:B21`
- Метрики:
  - WoW расход, %
  - WoW заявки, %
  - WoW CPL, %
  - Доля лидов из A06
  - Доля бюджета в сегментах с 0 лидами
- Технический helper для 0-lead share:
  - `N5:P` (служебный, скрывать при необходимости).

## 11) Лист «Кампании» (актуальная формула)
- Основная формула: `Кампании!A2`.
- Принцип:
  - агрегирование по `adset`;
  - сортировка по расходу `order by sum(Col6) desc`;
  - округление рублёвых метрик после `QUERY` через `LET + ROUND`.
- Рублёвые колонки:
  - `G` `CPC, ₽` (gross)
  - `I` `CPL, ₽` (gross)
  - `J` `расход, ₽` (gross)

## 12) Пороги подсветки (gross)
- Светофор KPI:
  - зелёный `< 1320`
  - жёлтый `1320..1584`
  - красный `> 1584`
- Применение:
  - `Сводка!B8` (общий CPL)
  - `Сводка!G:G` (недельный CPL)
  - `Сводка!L:L` (месячный CPL)
- `Кампании`:
  - зелёный: `CPL <= 1320` и `лиды >= 3`
  - красный: `CPL > 1800` и `лиды >= 1`

## 13) Режим экономии Composio tool calls (обязательный)

### Целевой бюджет вызовов
- Норма на один цикл обновления: `2-5` calls.
- Рабочий потолок на один цикл: `<=10` calls.

### Разрешённый сценарий обновления
1. Локально распарсить файлы из `data/` (без calls к Sheets).
2. Очистить диапазон данных на листе `Данные` одним вызовом.
3. Записать весь 2D-массив в `Данные` одним batch-вызовом.
4. Опционально обновить `last_sync` одной ячейкой (если используется служебное поле).
5. Удалить обработанные файлы из `data/` (кроме `README.md`).
6. Не трогать графики/сводные/оформление в этом цикле.

### Что запрещено в обычной загрузке
- Построчная запись (`append row` в цикле).
- Поячейчные обновления.
- Повторное оформление листов при каждом импорте.
- Поиск таблицы по имени вместо фиксированного `spreadsheet_id`.
- Лишние чтения диапазонов «для подстраховки», если расчёт можно сделать локально.
- Хранить старые raw-выгрузки в `data/` после успешного импорта.

### Стратегия чтения данных
- По умолчанию читать обратно только контрольные точки:
  - факт успешной записи,
  - опционально 1-2 агрегата для smoke-check.
- Полный разбор KPI из Sheets читать только по отдельной задаче анализа.

### Инкрементальная модель (не по умолчанию)
- Допустима только при отдельной настройке дедупа.
- Ключ дедупа: минимум `date + campaign_id + adset_id + ad_id + level`.
- До внедрения дедупа по умолчанию используем `full refresh`.

## 14) Нейминг-контракт (человеко-читаемый, обязательный)

### Форматы имен в кабинете
1. `campaign_name`:
`VK | <тип> | Парная бачата | <месяц>`

2. `adset_name`:
`[Axx] <сегмент> | <пол+возраст> | НСК`

3. `ad_name`:
`[Cxx] <угол> | <формат> | <оффер> | vN`

### Что извлекаем в аналитике
- `audience_id` из `adset_name` по regex: `\\[(A\\d{2})\\]`
- `creative_id` из `ad_name` по regex: `\\[(C\\d{2})\\]`
- `creative_angle` (по тексту в ad_name):
  - `Боль`
  - `Результат`
  - `Соцдоказ`

### Политика совместимости
- Исторические названия (старого формата) не переименовываем задним числом.
- Новый стандарт обязателен для всех новых запусков и дубликатов с новой гипотезой.
